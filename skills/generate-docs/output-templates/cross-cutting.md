Example for `IsClientOwner`:

The permission class compares `request.user.client_id` to a `client_id` value resolved from either the URL kwarg or the request body. If they match, the request is allowed; otherwise, DRF raises 403.

It's used on every endpoint where the URL or body specifies a `client_id` and the resource being modified belongs to that client. The check happens before the view runs, which means the view can assume `request.user` owns the target resource.

There's a known footgun: if a new view takes a `client_id` URL kwarg but forgets to add `IsClientOwner` to its `permission_classes`, the check is silently absent. There's no automatic enforcement. We rely on code review to catch this.

</details>

<details>
<summary><b>ЁЯУМ When to use</b></summary>

- Whenever a view's URL or request body includes a `client_id` and the resource is client-scoped
- NOT for views scoped only by `request.user` (that's already implicit)

</details>

<details>
<summary><b>ЁЯУж Side effects</b></summary>

- None тАФ pure check; raises `PermissionDenied` (translates to 403) on mismatch

</details>
